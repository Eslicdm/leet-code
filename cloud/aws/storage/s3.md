# Amazon S3

S3 (Simple Storage Service) is object storage built for durability, scale, and HTTP access. It is the default choice for backups, logs, static assets, data lakes, and cross-service file exchange.

### Key Features

| Feature | Description |
| :-- | :-- |
| Buckets and objects | Store any number of objects up to 5 TB each. |
| Storage classes | Standard, Intelligent-Tiering, Standard-IA, One Zone-IA, Glacier tiers. |
| Versioning | Keep multiple versions of an object for recovery. |
| Lifecycle rules | Transition objects to colder tiers or expire them automatically. |
| Event notifications | Trigger Lambda, SQS, or SNS on object events. |
| Replication | Cross-region or same-region replication for durability and latency. |
| Pre-signed URLs | Time-limited URLs to grant temporary read or write access. |
| Strong consistency | Read-after-write consistency for all S3 operations since 2020. |

### When to Use

- You need durable object storage with HTTP access.
- You store backups, logs, static website assets, or media files.
- You build a data lake for analytics.
- You need a target for cross-region or cross-account replication.

### When Not to Use

- You need a mounted POSIX file system (use [EFS](https://aws.amazon.com/efs/) or FSx).
- You need block storage attached to a single EC2 instance (use [EBS](./ebs.md)).
- You need sub-millisecond key-value lookups with high write rates (consider [DynamoDB](../database/dynamodb.md)).

### Pricing Model

- Pay per GB-month of storage, requests, and data transfer out.
- Storage class and lifecycle rules strongly affect cost.
- No charge for data transfer in to S3 from most AWS services.

### Trade-offs

- Pros: 11 nines of durability, unlimited scale, deep AWS integration, rich security controls.
- Cons: eventual consistency is a thing of the past, but object listings and prefix queries can be expensive at high scale.

### Practical Tips

- Use lifecycle rules to move old data to Glacier or expire it.
- Enable versioning and MFA Delete for critical buckets.
- Use S3 Object Lock for compliance and immutability.
- Use S3 Access Points to give each application a tailored view of a shared bucket.
- Use bucket policies plus IAM policies plus Block Public Access; never rely on a single layer.
